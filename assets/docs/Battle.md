# Battle

### 對戰機制設計文件

---

## 核心特性

* **回合制**：戰鬥以回合為單位進行，雙方輪流執行行動。
* **1對1對戰**：戰鬥由兩名角色進行，不涉及隊伍或多人戰鬥。
* **公平規則**：對戰雙方在規則上完全公平，雙方享有相同的機制、權限和限制。
* **單機和多人支援**：架構支援單機遊戲和Steam/IP網路對戰。

---

## 對戰準備流程

在進入實際戰鬥迴圈前，雙方玩家需經歷**5 個準備步驟**，每個步驟對方都無法看到其選擇，最大化對戰的不確定性：

### 準備步驟 1：角色選擇

* 玩家從可用角色中選擇一個作為該場對戰的使用角色
* 對手的角色選擇在此時**不可見**
* 可在戰鬥過程中觀察對手的角色能力來推斷其身份

### 準備步驟 2：被動特質選擇

* 玩家從被動特質庫中選擇 2-4 個特質作為本場對戰的風格
* 被動特質包括傷害加成、減傷、資源優化、狀態強化等多種類型
* 對手的被動特質選擇**不可見**，玩家需在對戰中通過觀察推斷
* 詳見 [PassiveTrait.md](./PassiveTrait.md) 的完整特質列表

### 準備步驟 3：行動配置

* 玩家從角色的**可用行動集合**中進行選擇配置：
  * **角色專屬行動**：從角色獨有的招式中選擇固定數量（例如 4 個）
  * **共通行動**：從全角色通用的行動中選擇固定數量（例如「攻擊」、「防守」、「休息」各 1 個）
* 此步驟允許玩家**客製化角色的行動庫**，同一角色在不同對戰中可有不同的招式組合
* 對手的行動配置**不可見**，在對戰中觀察對手使用的招式來反推其配置

### 準備步驟 4：神恩選擇

* 雙方從隨機挑選的神明中選擇一位作為該場試煉的庇佑者
  * 系統隨機生成 5 個不同勢力的神明，附加預設角色的主保護神（如有）
  * 玩家可看到**所有可用神明的名稱**（兩方相同），但**看不到對手的選擇**
* 在各自選擇的神明下，玩家需從 3 個難度選項中挑選任務：
  * 🟢 **簡單**：較低要求，較小獎勵
  * 🟡 **標準**：平衡要求和獎勵
  * 🔴 **困難**：高難度挑戰，豐厚獎勵
* 玩家**看不到對手的任務難度選擇和具體內容**
* 任務的完成與否完全由玩家的**實際行動**在戰鬥中決定
* 詳見 [DivineFavor.md](./DivineFavor.md) 的完整神明與任務庫

### 準備步驟 5：戰鬥開始

* 所有準備完畢，雙方進入實際對戰
* 玩家視角只知道對手的**角色**，對於被動特質、行動配置、神明青睞等資訊**完全盲目**
* 玩家需根據對手的行動、傷害值、恢復速度等細節推斷其風格

## 對戰迴圈流程

對戰的基本流程分為四個主要階段：

### 1. 回合開始（Turn Start）

* 檢查所有角色的**效果與異常**，觸發回合開始時的事件（例如「燃燒」狀態造成傷害）。
* 減少所有**效果與異常**的持續時間。
* 減少所有**動作**的冷卻時間。
* 檢查**姿態**的持續時間，並清除已過期的姿態。
* 恢復角色的**耐力值**（可選）。

### 2. 行動選擇（Action Selection）

* 雙方角色各自選擇該回合的行動（Action）。
* 根據角色的當前**姿態**，動作選擇介面會動態更新，只顯示該姿態下允許的動作。
* 選擇過程可以是**玩家操控**或**AI決策**，取決於角色的類型。
* 雙方必須同時或以預設順序提交其行動選擇。

### 3. 結果應用（Result Application）

* **行動解析**：按照預設的執行順序（基於敏捷度AGI的機率性先手），解析雙方的行動。
* **被動特質應用**：應用雙方已選擇的被動特質對傷害、命中、恢復等的修正。
* **神恩加護應用**：應用已完成任務的神恩加護效果對本回合的計算結果進行修正。
* **任務進度更新**：根據本回合的行動更新玩家的神恩任務進度。
* **任務完成檢測**：如果玩家完成了神恩任務，立即通知對手並顯示獲得的加護效果。
* **效果計算**：對每個行動進行計算，包括：
    * **傷害計算**：根據動作的固定傷害值和角色被動特質的加成/減傷計算最終傷害。
    * **命中判定**：根據動作的固定命中率和角色被動特質的修正判定行動是否命中。
    * **特殊效果**：應用行動所帶來的**狀態效果**、**姿態改變**或其他機制互動。
* **數值變更**：更新相關角色的戰鬥參數（HP、MP、STA 等）。
* **環境變化**：如果戰鬥環境本身也是可變的（例如天氣、地形等），也在此階段進行更新。

### 4. 回合結束（Turn End）

* 檢查戰鬥是否已結束（例如檢查任何一方的 HP 是否歸零）。
* 執行回合結束時觸發的事件（例如「中毒」狀態造成傷害）。
* 如果戰鬥未結束，回到「回合開始」階段以開始下一回合。
* 如果戰鬥已結束，顯示戰鬥結果（勝利、失敗或其他結局）。

## 系統架構（v2.0）

### 分層架構

對戰系統採用**三層架構**，確保單機和多人模式都能使用：

```
┌─────────────────────────────────────────┐
│  UI 層 (Battle.gd)                      │
│  - 顯示狀態、播放動畫、處理玩家輸入      │
└─────────────┬───────────────────────────┘
              │ 信號和方法呼叫
┌─────────────▼───────────────────────────┐
│  對戰管理層                              │
│  - BattleManager (單機/基類)             │
│  - ServerBattleManager (伺服器)          │
│  - ClientBattleManager (客戶端)          │
└─────────────┬───────────────────────────┘
              │ 使用
┌─────────────▼───────────────────────────┐
│  邏輯層 (BattleLogic + BattleState)      │
│  - 無狀態計算函數                        │
│  - 狀態序列化和驗證                      │
└─────────────────────────────────────────┘
```

### 核心組件

**BattleLogic（無狀態邏輯層）**
- 純計算函數，支援確定性計算
- 用於伺服器驗證和客戶端預測
- 不儲存任何狀態

**BattleState（狀態序列化層）**
- 儲存當前對戰狀態（HP、MP、STA、冷卻）
- 可序列化為字典用於網路傳輸
- 提供狀態簽名用於驗證

**BattleManager（基礎對戰管理）**
- 核心回合流程管理
- 預設為單機模式（MODE_SINGLEPLAYER）
- 現有代碼完全相容

詳見 [`BattleSystemArchitecture.md`](./BattleSystemArchitecture.md) 以瞭解完整的多人實現指南。

## 行動執行順序

在「結果應用」階段，行動的執行順序至關重要：

### 順序規則

1. **機率性先手**：根據雙方的**敏捷（AGI）**計算先手機率。
    * 先手機率 = 50% + (己方AGI - 敵方AGI) × 2%
    * 高敏捷值意味著更高的先手機率，但不保證永遠先手。
    * 某些行動可能帶有額外的優先度修正，例如「反擊」類行動可能擁有更高的優先度。
2. **同速並發**：如果雙方在該回合的執行順序相同，雙方行動將**同時發生**，互不影響執行順序。
    * 例如，兩個敏捷值相同的角色同時發動攻擊，各自的傷害都會正常計算並應用。

### 互動影響

* **姿態改變的影響**：如果A角色在此回合改變了自己的姿態（例如從「站立」變為「防禦」），這不會影響本回閤中B角色已經選定的行動的結果。
* **參數變化的影響**：如果A角色在此回合降低了自己的防禦力，這不會影響本回閤中B角色已經選定的傷害計算（傷害已經在行動選擇時根據當時的參數預先計算）。

> **注意**：在純粹的同時行動制中，所有計算都基於行動選擇時的角色狀態，而不是實時狀態。這確保了雙方的完全公平性。

## 勝利與失敗條件

* **勝利**：對手的 HP 歸零或滿足特殊勝利條件。
* **失敗**：自身的 HP 歸零或無法繼續戰鬥。
* **平局**（可選）：在特定回合數後，如果雙方都未分出勝負，可判定為平局。

## 遊戲設計考量

### 公平性保證

* 雙方享有相同的**屬性計算公式**。
* 雙方享有相同的**行動庫**（即兩個角色都能使用相同類型的動作）。
* 雙方享有相同的**效果與異常規則**。
* 雙方在**行動選擇**時都能看到完全相同的信息（如對手的當前 HP、姿態等）。

### 策略深度

* **動作選擇**：玩家需要考慮對手可能的行動，並相應地選擇最佳應對方案。
* **資源管理**：MP、STA 等資源的有限性迫使玩家做出取捨。
* **冷卻時間機制**：防止玩家無腦重複使用最強動作。
* **姿態系統**：為戰鬥增添了戰略層面，玩家需要在時機恰當時切換姿態。
* **效果與異常**：通過狀態的疊加和相互作用，創造出更複雜的戰鬥場景。

## 網路同步（多人模式）

### 模式切換

```gdscript
# 單機模式（預設，現有代碼相容）
var battle_manager: BattleManager = $BattleManager
battle_manager.battle_mode = BattleManager.MODE_SINGLEPLAYER

# 伺服器模式（主機）
var battle_manager: ServerBattleManager = $BattleManager
battle_manager.setup_multiplayer(player1_peer_id, player2_peer_id)

# 客戶端模式（遠端玩家）
var battle_manager: ClientBattleManager = $BattleManager
battle_manager.setup_client_connection(local_player_id, server_peer_id)
```

### 反作弊保證

1. **伺服器計算**：所有傷害、命中、狀態效果都在伺服器計算
2. **動作驗證**：伺服器驗證客戶端提交的動作是否合法
3. **狀態驗證**：定期檢查客戶端狀態是否與伺服器一致
4. **確定性 RNG**：使用伺服器種子確保隨機數無法被篡改

### 樂觀更新

客戶端即時顯示玩家選擇，但等待伺服器確認：
- 提交動作後立即更新 UI
- 接收伺服器結果後驗證和同步
- 如果出現不一致，強制覆蓋為伺服器狀態

## 補充說明

* **同時行動制**：採用完全同時的行動制，確保公平性，但要求玩家在不知道對手選擇的情況下做出決策。
* **可視化反饋**：需要通過清晰的視覺效果和動畫來展示戰鬥過程。
* **擴展性**：此框架可支持多人戰鬥、隊伍制或其他複雜戰鬥模式。

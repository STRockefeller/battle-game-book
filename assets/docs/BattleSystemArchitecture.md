# 對戰系統架構文檔

## 概述

重構後的對戰系統支援單機和多人模式，通過分離邏輯層、狀態層和UI層，實現了易於擴展和維護的架構。

## 核心組件

### 1. 邏輯層 (BattleLogic)

- **職責**: 純計算函數，決定遊戲規則如何執行
- **特點**: 無狀態、完全確定性、用於伺服器驗證和客戶端預測
- **主要計算**:
  - 命中判定（基於動作固定命中率和被動加成）
  - 傷害計算（動作固定傷害 × 被動傷害加成% × (1 - 被動減傷%)）
  - 執行順序（基於AGI的機率性先手，公式：50% + (己方AGI - 敵方AGI) × 2%）
  - 動作驗證（資源足夠、冷卻檢查、姿態允許）
  - 戰鬥結束檢查（任一角色血量 ≤ 0）

**設計特點**：
- 純函數，沒有副作用
- 相同輸入必定產生相同輸出
- 支援單機和多人驗證

### 2. 狀態層 (BattleState)

- **職責**: 管理對戰狀態的序列化和驗證
- **內容**: HP/MP/STA、冷卻計時、角色屬性快照
- **特點**: 可轉換為字典便於網路傳輸、可生成狀態簽名用於驗證

**設計特點**：
- 完全序列化，支援發送和保存
- 提供狀態哈希用於多人同步驗證

### 3. 對戰管理層 (BattleManager)

- **職責**: 協調整個對戰流程
- **模式**:
  - **單機**（BattleManager）：直接控制 AI，無網路
  - **伺服器**（ServerBattleManager）：驗證、計算、廣播
  - **客戶端**（ClientBattleManager）：樂觀更新、顯示結果

**設計特點**：
- 信號系統通知 UI 層狀態變化
- 三層架構支援單機→多人的平滑升級

## 對戰流程

### 單機模式流程

```
1. 回合開始 → 觸發事件、減少冷卻
   ↓
2. 行動選擇 → 玩家和 AI 同時選擇
   ↓
3. 結果應用 → 按優先度執行、計算傷害、應用效果
   ↓
4. 回合結束 → 檢查戰鬥是否結束
   ↓
重複至戰鬥結束
```

**關鍵特性**：
- 同時行動制：雙方同時選擇，根據敏捷度決定實際執行順序
- AI 自動決策：無需玩家干預
- 動作效果：傷害、冷卻、狀態效果同時應用

### 多人模式流程（簡化）

```
單機邏輯 + 網路層：

1. 玩家提交動作 → 發送到伺服器
   ↓
2. 伺服器驗證 + 計算 → 決定結果
   ↓
3. 伺服器廣播結果 → 所有客戶端同步
   ↓
重複至戰鬥結束
```

**關鍵特性**：
- 伺服器計算驗證（反作弊）
- 客戶端樂觀更新（響應快）
- 狀態同步驗證（防止差異）

## 反作弊原則

### 1. 伺服器驗證

所有關鍵計算僅在伺服器執行，客戶端無法修改傷害、命中、冷卻和資源扣除。

### 2. 動作驗證

伺服器驗證每個動作的合法性，拒絕非法動作（資源不足、冷卻未清、姿態不允許）。

### 3. 狀態簽名

定期驗證客戶端狀態與伺服器一致，檢測到差異時強制客戶端同步伺服器狀態。

### 4. 確定性隨機

使用伺服器種子計數器，客戶端無法預測或篡改隨機結果。

## 設計要點

### 單機 ↔ 多人平滑升級

架構設計允許從單機逐步擴展到多人而無需大規模重構：
- 單機：直接使用 BattleManager
- 多人：替換為 ServerBattleManager/ClientBattleManager
- 邏輯層完全共享（BattleLogic 用於客戶端預測和伺服器驗證）

### 同時行動制遊戲感

雙方同時選擇行動但根據優先度（敏捷度）順序執行，提升策略性：
- 玩家無法根據對手行動被動應對
- 優先度系統獎勵敏捷力量分配

### 信號系統 UI 解耦

使用信號讓邏輯層獨立於 UI，便於測試和擴展：
- `turn_start_selection` - UI 更新可用動作按鈕
- `action_executed` - UI 播放動作動畫和傷害數字
- `turn_ended` - UI 記錄戰鬥日誌
- `battle_ended` - UI 顯示勝負畫面
